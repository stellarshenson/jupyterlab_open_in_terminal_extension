6. **Task - Add file and empty area support** (v1.0.9): Extended context menu to support files and empty area in file browser<br>
   **Result**: Major feature enhancement to support three context scenarios. Added `PathExt` import from `@jupyterlab/coreutils` to extract parent directory from file paths. Updated execute function to handle: (1) folders - opens terminal at folder path, (2) files - uses `PathExt.dirname()` to open terminal in parent folder, (3) empty area - uses `fileBrowser.model.path` for current directory. Changed `isVisible` to always return true since command works in all contexts. Consolidated from multiple selectors (`.jp-DirListing-item[data-isdir="true"]`, `.jp-DirListing-item[data-isdir="false"]`) to single `.jp-DirListing-content` selector to eliminate duplicate menu items that appeared when clicking on items inside the content area. Renamed menu label from "Open in Terminal" to "Open Location in Terminal" for clarity. Updated `package.json` with `@jupyterlab/coreutils` dependency. Updated README and unit tests to reflect new behavior. Extension now provides single unified context menu item that intelligently handles all file browser click targets.

7. **Task - Suppress duplicate core menu item** (v1.0.11): Fixed duplicate "Open in Terminal" appearing on folders alongside our "Open Location in Terminal"<br>
   **Result**: Root cause traced to JupyterLab core, not our code. JupyterLab 4.6 ships the built-in plugin `@jupyterlab/terminal-extension:open-folder-in-terminal` (description: 'Adds an "Open in Terminal" command for the file browser.') which registers command `terminal:open-folder-in-terminal` with label "Open in Terminal" and a folder-only context menu item. Our extension's `.jp-DirListing-content` item ("Open Location in Terminal") is a superset that also covers files and empty area, so on folders both appeared. Verified our installed bundle contained only "Open Location in Terminal" while the plain label lived in core's `jlab_core` bundle. Fixed by declaring `disabledExtensions: ["@jupyterlab/terminal-extension:open-folder-in-terminal"]` under the `jupyterlab` metadata in `package.json`. Confirmed via `jupyterlab_server/config.py` that a federated (prebuilt) extension's `disabledExtensions` metadata is consumed and merged into the app page config on install. Verified after `make install` that the installed labextension `package.json` carries the field and the assembled page config lists the core plugin as disabled. Fix is self-contained (ships with the package), reversible (uninstall restores core), and requires no environment-level `jupyter labextension disable`.

8. **Task - Reconcile with stock plugin** (v1.0.14): Replaced the disable-core approach with cooperation - complement the native command instead of suppressing it<br>
   **Result**: The v1.0.11 fix disabled core's `terminal:open-folder-in-terminal`, leaving a divergent parallel implementation. Reconciled instead: removed `disabledExtensions` from `package.json` so core owns folders again. In `src/index.ts` restored the standard label "Open in Terminal", reused core's `terminal:create-new` command with `{cwd}` exactly as the stock folder command does, added the `serviceManager.terminals.isAvailable()` guard, and bound `terminalIcon` with the `menuItem` stylesheet to match native rendering. `isVisible` now returns false when a directory is selected, so the single `.jp-DirListing-content` item shows only for files (parent folder) and the empty area (current folder) - core stays unique on folders, no duplicate. Updated README, package description, and unit tests. Verified: lint clean, 5 tests pass, `make install` gives one item per context.
